# Reference-Grounded Generation

This directory holds the inputs for provenance-first candidate generation.

## Design

The pipeline is intentionally conservative:

- every candidate is tied to a curated source in `source_catalog.yaml`
- every candidate begins as a structured seed in `section_seed_specs.yaml`
- generated output goes to `benchmark/generated-candidates/`, not the canonical `benchmark/queries/`
- each generated package includes `query.yaml`, `rubric.yaml`, `solution.yaml`, and `provenance.yaml`

The benchmark standard is not merely "hard mathematics". Generated queries are
supposed to target `research-style pure mathematical reasoning` and should
resemble the sorts of questions that advanced graduate students, PhD students,
postdocs, and working mathematicians might actually ask an LLM during research.

## Why separate candidates?

Generated items are not automatically benchmark items. They remain `draft`
candidates until:

1. an area expert confirms the mathematical soundness of the query
2. an area expert confirms the rubric admits standard valid approaches
3. an editor reviews contamination risk, clarity, and MSC placement

## Generation methodology

Every candidate should be generated by the following workflow:

1. start from an authoritative source recorded in `source_catalog.yaml`
2. extract a theorem-level or concept-level seed suitable for benchmarking
3. adapt that seed into a research-style query rather than copying a routine exercise
4. assign official MSC placement and owning-area metadata
5. generate the query, rubric, solution outline, and provenance record together
6. reject items that are too shallow, too contest-like, missing hypotheses, or unsupported by the cited source

Preferred query families:

- theorem applicability
- proof strategy
- hypothesis sensitivity
- classification
- counterexample
- framework comparison
- invariant interpretation

Anti-patterns to reject:

- definition-only prompts
- reference-request prompts
- one-line textbook drills
- generic olympiad/contest stand-ins for research mathematics
- unsupported model-invented mathematical claims

## Promotion workflow

Candidate packages move into the canonical benchmark only after:

1. area review of query soundness and MSC placement
2. rubric review in the same field
3. editorial adjudication of clarity, contamination risk, and final status

Until then, they should remain in `benchmark/generated-candidates/`.

## Running the pipeline

Use:

```bash
npm run candidates:generate
```

This regenerates the entire `benchmark/generated-candidates/` tree from the
current source catalog and seed batch.
